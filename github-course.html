模块一：寻找 & 获取项目

1第一节：认识 GitHub

本节不讲复杂命令，只建立直觉。你先理解 GitHub 是一个公开项目平台，而不是普通网盘。

你需要先理解

可以这样理解 GitHub：

“GitHub 就像一个公开的代码图书馆。全世界的开发者会把项目放在这里，有些项目是完整软件，有些是网页模板，有些是文档、规范、案例库。我们今天不会一上来写复杂代码，而是先学会怎么看懂这个图书馆。”

然后解释两个概念：

  • Git：记录文件修改历史的工具，类似更强大的版本记录。
  • GitHub：放 Git 项目的云端网站，也方便展示、分享和协作。

操作案例：openclaw/openclaw

  1. 打开 openclaw/openclaw。
  2. 找到页面顶部的作者名、仓库名和分支。
  3. 找到绿色的 Code 按钮。
  4. 观察页面里的文件夹、文件和 README，理解一个 GitHub 仓库通常由很多部分组成。
本节重点GitHub 是项目主页，不只是下载页面。
暂时不用掌握分支、PR、Actions 等概念后面再了解。

2第二节：搜索好项目

本节训练“怎么找”。重点不是写代码，而是学会用关键词和项目指标做判断。

判断方法

“在 GitHub 上搜索项目，不能只看标题。我们要看它有多少人收藏、最近是否更新、README 是否清楚、有没有 License。零基础阶段，先学会判断一个项目值不值得打开。”

可以解释这些指标：

  • Stars：类似收藏数，通常能反映受欢迎程度。
  • Topics：项目标签，帮助我们按主题找资源。
  • Last commit：最近更新时间，用来判断是否还在维护。
  • README：项目说明书，下载前先读这里。

操作任务

本节使用两个搜索目标：

  • VoltAgent/awesome-design-md：练习搜索集合型资源。
  • anthropics/skills：练习搜索具体 AI Skills 仓库。

在 GitHub 搜索框里依次尝试下面的关键词：

awesome design md
DESIGN.md
agent skills
github skills
anthropic skills
language:markdown stars:>100

观察搜索结果里的仓库简介、Star 数、更新时间和 README 完整度。找到 anthropics/skills 后，重点看它是否是官方组织发布、README 是否清楚、目录是否有规律。

常见误区

Star 多不等于一定适合自己。零基础阶段要优先选择 README 清楚、结构简单、目标明确的项目。

3第三节：读懂项目主页

本节训练信息提取能力，让你知道下载前先看什么。

页面区域 是什么意思 你应该怎么看
README 项目说明书 先读这里，判断项目用途、安装方式、使用限制。
Stars / Forks 收藏数和复制分支数 辅助判断受欢迎程度，不作为唯一标准。
Issues 问题反馈区 看看别人是否遇到很多无法解决的问题。
License 使用授权 判断能不能复制、修改、商用或公开发布。
Last commit 最近更新时间 太久没更新的项目要谨慎使用。

  2. 它更像代码项目，还是资源集合？
  3. 应该先看 README 还是直接下载？
  4. 如果要复用其中一个 DESIGN.md，是否必须下载整个仓库？

4第四节：把项目下载到本地

本节理解 ZIP 下载和 git clone 的区别，同时理解仓库、文件夹、文件的层级关系。

两种下载方式

“GitHub 上的绿色 Code 按钮提供两种常见获取方式。Download ZIP 像是把整本书复印一份带走，简单但后续和原项目断开。git clone 像是把书带回家并保留和图书馆的连接，以后可以同步新版。”

方式 适合情况 注意事项
Download ZIP 只想看看、复制某些文件 后续更新要重新下载。
git clone 长期学习、修改、跟进项目 需要本地安装 Git。

课堂案例：vercel-labs/agent-skills

打开 vercel-labs/agent-skills/tree/main/skills。

你需要分清：

  • vercel-labs/agent-skills 是仓库。
  • skills 是仓库里的一个文件夹。
  • 点击 Download ZIP 通常下载的是整个仓库，不只是当前文件夹。
git clone https://github.com/vercel-labs/agent-skills.git

常见错误

初学者经常以为“我现在看的是 skills 文件夹，所以下载时只会下载 skills 文件夹”。实际使用时要注意：GitHub 的下载通常以仓库为单位。

模块二：上传 & 部署项目

模块目标：创建自己的网页项目，把它上传到 GitHub，并通过 EdgeOne Pages 部署成可访问的网站。

5第五节：Git 是什么

本节用“游戏存档”理解 Git，不陷入术语。

Git 的直觉

“打游戏时，在重要关卡前你会存档。Git 做的事很像：每次我们改完一批文件，就手动保存一个版本。以后如果改错了，可以回到之前的版本。”

这节开始创建课堂项目：my-agent-resource-page。它是一个资源收藏网页，展示今天学过的 3 个 GitHub 仓库。

命令类比

Git 命令 类比 实际作用
git add 把要保存的东西放进存档袋 选择本次要记录的文件。
git commit 按下存档键并写备注 正式保存一个版本。
git push 把存档上传到云端 同步到 GitHub。

6第六节：创建第一个仓库

本节创建属于你自己的 GitHub 仓库，为上传和部署做准备。

项目结构

在本地创建文件夹：

my-agent-resource-page/
├── index.html
├── style.css
└── script.js

页面主题：我收藏的 3 个 GitHub AI 资源。

GitHub 创建仓库步骤

  1. 登录 GitHub。
  2. 点击右上角 +，选择 New repository。
  3. 仓库名填写 my-agent-resource-page。
  4. 选择 Public，方便后续 EdgeOne Pages 部署。
  5. 先不要勾选自动创建 README，避免和本地项目冲突。

命名建议

仓库名建议使用英文小写、数字和短横线，例如 my-first-site。不要使用中文、空格或特殊符号。

7第七节：三步走 add / commit / push

本节把本地项目上传到 GitHub，并通过一次修改练习巩固流程。

首次上传命令

cd my-agent-resource-page

git init
git branch -M main
git remote add origin https://github.com/你的用户名/my-agent-resource-page.git

git add .
git commit -m "add my GitHub resource page"
git push -u origin main

课堂练习

  1. 打开 index.html。
  2. 把页面标题改成“我的 GitHub AI 资源收藏”。
  3. 重新执行 git add .。
  4. 执行 git commit -m "update page title"。
  5. 执行 git push。
  6. 刷新 GitHub 仓库页面，观察文件是否更新。

常见错误排查

  • 提示不是 Git 仓库：确认是否进入了正确文件夹。
  • push 失败：检查远程仓库地址是否写错。
  • GitHub 没变化：确认是否执行了 commit 和 push。

8第八节：EdgeOne Pages 一键部署

本节把 GitHub 仓库部署成公网网站，让你获得一个可以分享的成果链接。

部署理解

“GitHub 负责保存项目，EdgeOne Pages 负责把网页发布到互联网上。它们连接以后，每次我们 push 到 GitHub，EdgeOne Pages 都会自动重新部署。”

  1. 打开 EdgeOne Pages 并登录。
  2. 选择通过 AI 对话或 Skill 的方式部署项目。
  3. 参考官方文档：使用 Skill 一句话部署项目。
  4. 连接 GitHub，选择 my-agent-resource-page。
  5. 确认配置后开始部署，等待生成公网访问链接。

部署后验证

  1. 打开 EdgeOne Pages 生成的网址。
  2. 确认页面能看到 3 个 GitHub 资源卡片。
  3. 回到本地修改一个介绍文案。
  4. 执行 git add .、git commit、git push。
  5. 等待 EdgeOne Pages 自动部署，刷新网站查看变化。
